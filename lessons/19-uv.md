## Introduction

### What this lesson is about

`uv` is a modern, very fast Python package manager and project tool written in
Rust.  It replaces separate tools like `pip`, `venv`, and `pip-tools` with a
single command.  This lesson covers installing `uv`, creating a project with a
virtual environment, adding packages, and understanding `requirements.txt` —
the older format you will still encounter in existing projects.

### Why you need this

Previous lessons used `pip3 install` to add packages globally.  This works for
experimenting, but in a real project you want each project to have its own
isolated set of packages at specific versions so that different projects don't
interfere with each other.  `uv` makes this fast and straightforward.  You
will use `uv` for the remaining lessons in this course.

## Do

### Step 1 — Install uv

### Mac / Linux:

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

### Windows (PowerShell):

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

Restart your terminal, then verify:

```bash
uv --version
# uv 0.x.x
```

---

### Step 2 — Create a new project

```bash
cd ~/projects
uv init my-project
cd my-project
```

`uv init` creates:
- `pyproject.toml` — project metadata and dependency list
- `.venv/` — a virtual environment containing only the packages you specify
- `hello.py` — a starter script

---

### Step 3 — Add a package

```bash
uv add requests
```

`uv` resolves, downloads, and installs `requests` and its dependencies into
`.venv/`.  It also records the exact version in `uv.lock` (a *lock file* that
ensures reproducibility) and adds a loose version constraint to
`pyproject.toml`.

---

### Step 4 — Run a script

```bash
uv run python hello.py
```

`uv run` automatically activates the virtual environment before running the
script.  You can also activate it manually:

### Mac / Linux:

```bash
source .venv/bin/activate
python hello.py
deactivate
```

### Windows:

```bash
.venv\Scripts\activate
python hello.py
deactivate
```

---

### Step 5 — Installing multiple packages

```bash
uv add pandas openpyxl matplotlib pytest
```

All packages are isolated to this project and recorded in `pyproject.toml`.

---

### Step 6 — requirements.txt (the older approach)

Many existing Python projects list their dependencies in a plain text file
called `requirements.txt`:

```
requests==2.31.0
pandas>=2.0
openpyxl
```

You may encounter this format in tutorials, GitHub repos, or at work.  With
`uv` you can install everything in a `requirements.txt` file with:

```bash
uv pip install -r requirements.txt
```

Or you can convert it to `uv`'s format by running `uv add` for each package.
The `requirements.txt` format is simpler but less powerful than
`pyproject.toml`: it doesn't track sub-dependencies or the Python version.

> **When you will see requirements.txt:** older courses, StackOverflow answers,
> and many production projects.  Knowing how to use it is important even though
> modern projects prefer `pyproject.toml`.

---

### Step 7 — Sharing your project

Because `uv.lock` records exact versions, anyone who clones your repository
can recreate the identical environment with:

```bash
uv sync
```

This is more reliable than `pip install -r requirements.txt` because `uv sync`
installs exact pinned versions rather than potentially picking newer ones.

## Explore

1. Create a new `uv` project, add `rich` (a library for colourful terminal
   output), and write a short script that uses it.  What version was installed?
2. What does `uv remove <package>` do?  Try removing a package and observe
   the change to `pyproject.toml`.
3. What is a *virtual environment*?  Why do we need one per project instead of
   one global installation?
4. Research the difference between `uv add` (which records in `pyproject.toml`)
   and `uv pip install` (which installs without recording).  When would you
   use each?
5. Run `uv tree` to see the full dependency tree of your project.  Why might a
   package you didn't explicitly request appear in the tree?
